This document describes the **wire protocol** used by Mighty camera streams and command/control RPC.

For SDK onboarding and app-level usage examples, see [index.mdx](./index.mdx).

## 1. Packet Framing

Each frame on the wire is:

```text
[4B header][4B type][4B payload_len][payload][4B crc32][4B footer]
```

- Header magic: `0xDEADBEEF`
- Footer magic: `0xFEEDFACE`
- Endianness: **big-endian** for numeric fields
- CRC: CRC32 of payload bytes (zero when payload is empty)

Raw-frame CRC compatibility rule:
- For `RAW ` and `SRAW`, decoders may accept `crc32 == 0` as a skip-CRC fast path (for device interoperability).

## 2. Type Codes

4-byte ASCII type values:

- `JPG `: JPEG image
- `RJPG`: reference JPEG image
- `RAW `: mono raw image
- `SRAW`: stereo raw image (left + right)
- `POSE`: canonical public body pose (`T_odom_body`)
- `UPOS`: reserved/internal pose stream (not part of public API)
- `LCON`: loop constraints / line segments
- `VIZ `: visualization payload (subtyped)
- `IMU `: IMU samples batch
- `STAT`: textual status
- `VSTA`: structured VIO status
- `RSET`: reset event
- `FEA3`: 3D features
- `PCLD`: point cloud
- `KEYF`: keyframe descriptor event
- `CMD `: command request
- `CRES`: command response
- `CFGQ`: config request payload
- `CFGR`: config response payload

## 3. Payload Layouts (High-Level)

The canonical builders/decoders are implemented in:
- JS: `js/core/protocol.js`
- C++: `cpp/mighty_protocol.h`
- Python: `python/mighty_protocol.py`

### 3.1 Images

#### `JPG ` / `RJPG`
- timestamp (u64)
- channel name length + channel bytes (`RJPG` may omit/normalize channel)
- JPEG bytes

#### `RAW `
- timestamp (u64)
- width (u32)
- height (u32)
- format (u8)
- channel name
- raw bytes

Formats (SDK constant: `RAW_FORMAT`):
- `UNKNOWN=0`
- `GRAY8=1`
- `RGB24=2`
- `BGR24=3`
- `RGBA32=4`
- `BGRA32=5`
- `YUV420SP=6`
- `YUV420P=7`

#### `SRAW`
- left raw image payload
- right raw image payload

### 3.2 Pose (`POSE`; `UPOS` reserved/internal)

`POSE` is the public pose stream.
`UPOS` keeps the same binary layout but is reserved for internal/debug usage.
For an integration-focused explanation and transform examples, see [Pose Guide](./pose.mdx).

Core fields:
- `poseType` (u32)
- `poseFlags` (u32)
- `positionM` (f64 x3)
- `confidence` (f32)
- optional fields gated by `poseFlags`

Public frame contract (`POSE`):
- `frame_id = "odom"` (ENU)
- `child_frame_id = "base_link"` (FLU)
- transform meaning: `T_odom_body`
- equation: `p_odom = R_odom_body * p_body + t_odom_body`
- `base_link` origin is the IMU origin

`poseType` semantics:
- `0`: body pose (`T_odom_body`) for public `POSE`
- `1`: camera pose (reserved/internal)
- other values: reserved

Quaternion convention:
- quaternion is source-to-parent (`q_odom_body` for public `POSE`)
- component order is `xyzw`

`poseFlags` bits:
- bit 0: quaternion present (`orientationXyzw`, f64 x4)
- bit 1: keyframe marker
- bit 2: linear velocity in body frame (`linearVelocityBodyMps`, f64 x3)
- bit 3: angular velocity in body frame (`angularVelocityBodyRps`, f64 x3)
- bit 4: linear acceleration in body frame (`linearAccelerationBodyMps2`, f64 x3)
- bit 5: angular acceleration in body frame (`angularAccelerationBodyRps2`, f64 x3)
- bit 6: `timestamp_ns` present (u64)

POSE invariants:
- `positionM`, `linearVelocityBodyMps`, `angularVelocityBodyRps`,
  `linearAccelerationBodyMps2`, and `angularAccelerationBodyRps2` are finite when present.
- quaternion is interpreted as `xyzw`; producers should send normalized quaternions.
- decoders clamp `confidence` into `[0, 1]`.
- `timestamp_ns` is an on-device monotonic timestamp when bit 6 is set.
- public contract fixes `frame_id="odom"` and `child_frame_id="base_link"` even though frame strings are not encoded in the wire payload.

Protocol evolution rule:
- append-only; new optional fields must be gated via new flags.

### 3.3 IMU (`IMU `)

Batch payload containing repeated samples:
- `timestamp_ns` (u64)
- accel `(ax, ay, az)` (f32)
- gyro `(gx, gy, gz)` (f32)

### 3.4 VIO State (`VSTA`)

Structured low-rate status telemetry.

Version 1 fields:
- `version` (u8)
- `state` (u8)
- `flags` (u16)
- `timestamp_ns` (u64)
- `fps_current` (f32)
- `fps_average` (f32)
- `pose_confidence01` (f32)
- `tracking_rate` (f32)
- `num_features` (u32)
- `loop_closures` (u32)

Version 2 appends:
- `build_version_len` (u8)
- `build_version` (bytes)

Version 3 appends:
- `imu_hz_current` (f32)
- `imu_hz_average_5s` (f32)

Version 4 appends:
- `init_reason_code` (u8)

Version 5 appends:
- `static_init_reason_code` (u8)
- `dynamic_init_reason_code` (u8)

### 3.5 Keyframe Descriptor (`KEYF`)

`KEYF` carries the timestamp and learned descriptor for a VIO keyframe. The
producer emits one packet only after the descriptor has been computed.

Payload layout, big-endian:

- `version` (u8), currently `1`
- `descriptor_type` (u8), currently `1` for float32
- `flags` (u16), currently `0`
- `timestamp_ns` (u64), matching the VIO/image timestamp of the keyframe
- `descriptor_dim` (u32)
- `descriptor` (`f32[descriptor_dim]`)

Version 6 appends:
- `memory_total_bytes` (u64)
- `memory_used_bytes` (u64)
- `memory_free_bytes` (u64)

Version 7 appends:
- `light_level01` (f32): producer-defined raw image-quality actual value; Mighty VIO on Rockchip currently sends an AE-derived darkness/exposure-stress proxy, or `NaN` when unavailable
- `light_required01` (f32): producer-defined raw requirement/limit value; Mighty VIO on Rockchip currently sends the darkness-proxy limit, or `NaN` when unavailable

Recommended state mapping:
- `0=OFF, 1=INITIALIZING, 2=TRACKING, 3=DEGRADED, 4=LOST, 5=LOW_LIGHT`

Recommended init reason mapping:
- `0=NONE`
- `1=WAITING_FOR_FIRST_IMU`
- `2=WAITING_FOR_INIT_FRAMES`
- `3=WAITING_FOR_PARALLAX`
- `4=WAITING_FOR_IMU_EXCITATION`
- `5=STATIC_INSUFFICIENT_FEATURES`
- `6=STATIC_SCENE_MOTION_TOO_HIGH`
- `7=RELATIVE_POSE_UNAVAILABLE`
- `8=GLOBAL_SFM_FAILED`
- `9=PNP_INSUFFICIENT_POINTS`
- `10=PNP_RANSAC_FAILED`
- `11=VISUAL_IMU_ALIGNMENT_FAILED`
- `12=UNKNOWN`
- `13=WAITING_FOR_FIRST_IMU_NO_SAMPLES`
- `14=WAITING_FOR_FIRST_IMU_NOT_YET_ALIGNED`
- `15=WAITING_FOR_FIRST_IMU_TIME_OFFSET_INVALID`

### 3.5 Status / Reset

#### `STAT`
- UTF-8 status text

#### `RSET`
- no payload

### 3.6 Viz / Constraints / Point Cloud

#### `VIZ `
- subtype-based payload:
  - `0`: features
  - `1`: detections
  - `2`: matches

#### `LCON`
- constraint segments (type + start/end 3D points)

#### `PCLD`
- points: `(x,y,z,r,g,b)`
- optional `point_size`

### 3.7 Command and Config

#### `CMD `
- `req_id` (u32)
- command name length + name bytes
- data length + data bytes

#### `CRES`
- `req_id` (u32)
- status (u8) (`0` success)
- message length + UTF-8 message
- data length + data bytes

Standard command names:
- `device_id` / `device-id`: returns the device's stable permanent identifier as UTF-8 text in both the `CRES` message and data fields.

#### `CFGQ`
- `version` (u8)
- `op` (u8): `0=GET`, `1=SET`
- key length + key bytes
- value length + value bytes

#### `CFGR`
- `version` (u8)
- `op` (u8)
- `success` (u8)
- `has_value` (u8)
- key length + key bytes
- message length (u16) + message bytes
- value length + value bytes

Config-over-command convention used by SDK:
- send `CMD(name="config", data=<CFGQ payload>)`
- receive `CRES(data=<CFGR payload>)`

## 4. Low-Level API Surface by Language

### JavaScript (`js/core/protocol.js`)

- framing: `makePacket`, `parseFrames`, `FrameDispatcher`
- builders: `build*Payload` helpers for each type
- decoders: `decode*Payload` helpers for each type
- constants: `TYPE`, `RAW_FORMAT`, `CONFIG_OP`

### Python (`python/mighty_protocol.py`)

- framing: `make_packet`, `parse_frames`, `FrameDispatcher`
- builders/decoders: type-specific functions matching protocol payloads
- constants: `TYPE`, `RAW_FORMAT`, `CONFIG_OP`

### C++ (`cpp/mighty_protocol.h`, `cpp/mighty_protocol_consumer.h`)

- framing: `make_packet`, `parse_frame`, `FrameConsumer`, `FrameDispatcher`
- builders/decoders: `build_*_payload`, `decode_*_payload`
- optional decoded callbacks: `DecodedDispatcher`

## 5. Compatibility Notes

- Preserve big-endian encoding across all languages.
- Keep frame types 4-byte ASCII.
- Extend payloads in append-only form where possible.
- Use explicit versioning (e.g., `VSTA`) when adding structured fields.
- Keep `CMD/CRES` request IDs unique per in-flight command to correlate responses.

SDK pose callback fields:
- JS SDK (`js/sdk/client.js`) pose callback object:
  - `isPublic`, `packetType`, `poseType`, `poseTypeRaw`, `poseFlags`
  - `frameId`, `childFrameId`
  - `positionM`, `orientationXyzw`
  - `linearVelocityBodyMps`, `angularVelocityBodyRps`
  - `linearAccelerationBodyMps2`, `angularAccelerationBodyRps2`
  - `timestampNs`, `confidence`, `isKeyframe`
- Python SDK (`python/mighty_sdk/client.py`) pose callback object:
  - `is_public`, `packet_type`, `pose_type`, `pose_type_raw`, `pose_flags`
  - `frame_id`, `child_frame_id`
  - `position_m`, `orientation_xyzw`
  - `linear_velocity_body_mps`, `angular_velocity_body_rps`
  - `linear_acceleration_body_mps2`, `angular_acceleration_body_rps2`
  - `timestamp_ns`, `confidence`, `is_keyframe`
- C++ SDK (`cpp/sdk/mighty_client.h`) `PoseFrame`:
  - `is_public`, `packet_type`, `pose_type`, `pose_type_raw`, `pose_flags`
